## Step 1: Study the External API

Before writing any code, deeply understand the external service.

### API Research Checklist

- [ ] Read the official API documentation end-to-end
- [ ] Identify the API version you will use (latest stable)
- [ ] Note authentication method (API key, OAuth2, JWT)
- [ ] Find rate limits (requests per second/minute, burst limits)
- [ ] Check data format (JSON, XML, form-encoded)
- [ ] Identify idempotency support (idempotency keys on POST)
- [ ] Review error response format and codes
- [ ] Check webhook support and event types
- [ ] Find sandbox/test environment availability
- [ ] Review SDKs вҖ” is there an official Node.js/Python SDK?
- [ ] Check SLA and uptime guarantees
- [ ] Review pricing вҖ” per-request costs, monthly limits

## Step 2: Authentication and Credential Management

### Auth Method Decision Tree

```
What auth does the external API use?
в”ңв”Җв”Җ API Key (static secret)
в”Ӯ   вҶ’ Store in environment variables (never in code)
в”Ӯ   вҶ’ Rotate keys quarterly
в”Ӯ   вҶ’ Use different keys for dev/staging/prod
в”Ӯ
в”ңв”Җв”Җ OAuth 2.0 (user-authorized)
в”Ӯ   вҶ’ Implement authorization code flow
в”Ӯ   вҶ’ Store refresh tokens encrypted in DB
в”Ӯ   вҶ’ Auto-refresh access tokens before expiry
в”Ӯ   вҶ’ Handle token revocation gracefully
в”Ӯ
в”ңв”Җв”Җ JWT / Bearer Token
в”Ӯ   вҶ’ Store securely (env var or secrets manager)

    вҶ’ Verify HMAC signature on every webhook

    вҶ’ Reject requests with invalid signatures

## Step 4: Error Handling and Rate Limits

### Error Handling Strategy

```
External API returns error
в”ңв”Җв”Җ 400 Bad Request
в”Ӯ   вҶ’ Log error details
в”Ӯ   вҶ’ Return user-friendly message
в”Ӯ   вҶ’ Do NOT retry
в”Ӯ
в”ңв”Җв”Җ 401 Unauthorized
в”Ӯ   вҶ’ Refresh token and retry once
в”Ӯ   вҶ’ If still 401 вҶ’ mark integration as disconnected
в”Ӯ   вҶ’ Notify admin
в”Ӯ
в”ңв”Җв”Җ 403 Forbidden
в”Ӯ   вҶ’ Log and alert вҖ” permissions may have changed
в”Ӯ   вҶ’ Do NOT retry
в”Ӯ
в”ңв”Җв”Җ 404 Not Found
в”Ӯ   вҶ’ Resource may have been deleted externally
в”Ӯ   вҶ’ Update local state to reflect
в”Ӯ   вҶ’ Do NOT retry
в”Ӯ
в”ңв”Җв”Җ 409 Conflict
в”Ӯ   вҶ’ Fetch current state and reconcile
в”Ӯ   вҶ’ Retry with updated data
в”Ӯ
в”ңв”Җв”Җ 429 Rate Limited
в”Ӯ   вҶ’ Read Retry-After header
в”Ӯ   вҶ’ Queue the request for later
в”Ӯ   вҶ’ Implement exponential backoff
в”Ӯ
в”ңв”Җв”Җ 500/502/503 Server Error
в”Ӯ   вҶ’ Retry with exponential backoff (max 3 retries)
в”Ӯ   вҶ’ Backoff: 1s вҶ’ 2s вҶ’ 4s (with jitter)
в”Ӯ   вҶ’ If all retries fail вҶ’ queue for manual retry
в”Ӯ   вҶ’ Alert on repeated failures
в”Ӯ
в””в”Җв”Җ Network Error (timeout, DNS failure)
    вҶ’ Retry with exponential backoff (max 3 retries)
    вҶ’ If all retries fail вҶ’ queue for manual retry
    вҶ’ Alert immediately
```

### Rate Limiting Strategy

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ  RATE LIMITING APPROACH                                      в”Ӯ
в”Ӯ                                                              в”Ӯ
в”Ӯ  1. KNOW THE LIMITS                                          в”Ӯ
в”Ӯ     вҶ’ Read API docs for exact limits                         в”Ӯ
в”Ӯ     вҶ’ Track: per-second, per-minute, daily, monthly          в”Ӯ
в”Ӯ     вҶ’ Note burst limits vs. sustained limits                 в”Ӯ
в”Ӯ                                                              в”Ӯ
в”Ӯ  2. IMPLEMENT CLIENT-SIDE THROTTLING                         в”Ӯ
в”Ӯ     вҶ’ Use a token bucket or leaky bucket algorithm           в”Ӯ
в”Ӯ     вҶ’ Set your limit to 80% of the provider's limit          в”Ӯ

в”Ӯ     вҶ’ Cache responses to reduce request volume               в”Ӯ
в”Ӯ                                                              в”Ӯ

в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

### Webhook Implementation Checklist

- [ ] Signature verification on every request (reject unsigned)
- [ ] Idempotency вҖ” store event IDs, skip duplicates
- [ ] Return 200 immediately вҖ” process async in background
- [ ] Store raw event payload for debugging and replay
- [ ] Handle out-of-order events (check timestamps/sequence)
- [ ] Dead letter queue for failed processing
- [ ] Retry mechanism for processing failures
- [ ] Monitoring dashboard showing event volume and failures
- [ ] Alert on spike in failures or unexpected event types

## Anti-Patterns вҖ” Never Do These

| Anti-Pattern | Why It Fails | Do Instead |
|-------------|-------------|-----------|
| Hardcode API keys in source code | Keys get committed and leaked | Use environment variables |
| Call external API from request handler | Slow responses, cascading timeouts | Use background jobs for slow calls |
| No retry on transient errors | Temporary failures become permanent | Retry with exponential backoff |
| Process webhooks synchronously | Slow processing causes timeouts and retries | Return 200 immediately, process async |
| No webhook signature verification | Anyone can send fake events | Always verify HMAC signature |
| No idempotency on webhook processing | Duplicate events cause duplicate actions | Track event IDs, skip duplicates |
| Tight coupling to provider's schema | Provider changes break your app | Use a mapper layer |
| No monitoring on integration health | Failures go unnoticed for hours | Dashboard + alerts from day one |
