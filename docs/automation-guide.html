Part 1 · Local / Internal Setup
Section 03 · Local Setup

Why Run Locally?

The local path keeps everything on your machine. Your code stays put. Your AI subscription does the work. There is no cloud quota, no team coordination, no centralized configuration — just you and the skills, available the moment you need them.

Local-only is the right starting point for solo developers, consultants working under NDAs, and small teams exploring whether agentic workflows are a fit before committing to broader rollout. Most engineering organizations begin here — not because they have to, but because it's the lowest-friction way to build trust with the tooling. Once your team feels comfortable, expanding to CI/CD becomes a natural next step rather than a leap of faith.
01
Privacy
Code never leaves your machine. No upload, no telemetry, no cloud-side processing. Ideal for regulated codebases and consulting work.
02
Cost
Uses your existing Claude or Cursor subscription. Zero infrastructure to provision. No CI minutes burned on advisory checks.
03
Speed
No CI startup delay. Skills run in seconds, not minutes. The feedback loop stays tight enough to keep you in flow.
04
Flexibility
Configure per developer. Each engineer chooses which routines to run, when, and where to send the output. No team coordination required.

Section 05 · Use Patterns

On-Demand vs Scheduled

There are two ways to use skills locally. Most developers start with on-demand and graduate to scheduled routines once the workflow becomes routine.

Pattern A · On-Demand

Run when you need it

You invoke the skill manually at the moment you want feedback. No scheduling, no automation — just typing the skill name when it makes sense in your flow. This is the simplest entry point and the right starting place for nearly every developer.

  • Before starting a small task — /quick-plan
  • To execute the plan — /quick-work
  • Before committing — /code-standards
  • Before opening a PR — /code-review
  • When stuck on a bug — /find-fix
Pattern B · Scheduled

Run on a routine

A scheduler wakes the runtime at predefined intervals — sleep, wake, run, post results, sleep. Useful for asynchronous review, weekly sweeps, and any work that benefits from a steady cadence rather than human triggering.

  • Every 4 hours — review your open PRs, post a digest
  • 3x per day — check recent commits for quality issues
  • Every Monday — run a security sweep on the codebase
  • Every Sunday night — generate a weekly engineering report
  • Quarterly — produce a compliance evidence package

These are the four routines most teams adopt. All cadences are configurable. All routines post to Google Chat as plain text by default — but Slack, Teams, email, or local file logs are equally supported.

Why plain text and not card-formatted messages? Plain text is readable in any chat client, paste-friendly into incident tickets, searchable in chat history, and survives platform migrations. Card formats break across clients, fragment when forwarded, and obscure the underlying content. We optimize for the format that still works five years from now.
Section 08 · Schedulers

Scheduling on Your Machine

Four ways to wake the runtime on a schedule. Pick by operating system and reliability needs — or skip local scheduling entirely and use a hosted equivalent. This page describes the options at a concept level. For implementation details, see the linked guides.

01
cron — Linux & macOS
The Unix standard. A crontab file lists when each routine runs. Editable from the terminal. Available on every Linux server and Mac out of the box. Lightweight, well-documented, universally supported.
Linux macOS Best for terminal users
02
launchd — macOS (recommended)
Apple's preferred scheduling system on Mac. Uses XML plist files. Much better than cron for laptop users — handles sleep and wake correctly, so a routine won't be skipped because the laptop was closed at the scheduled time.
macOS Best for laptops
03
Task Scheduler — Windows
The built-in Windows scheduler. GUI-based setup, no terminal required. Handles wake-from-sleep, missed runs, and conditional triggers. Best path for Windows-first developers and teams less comfortable with command-line config.
Windows GUI-driven
04
Hosted alternatives
If you'd rather not run scheduled jobs on your laptop: GitHub Actions schedule (free, runs in cloud — covered in Part 2), Cloudflare Workers Cron Triggers (free tier, serverless), or Vercel Cron Jobs (for teams already on Vercel).
Cloud No local setup
When you're ready to set this up, the implementation steps live in our CI/CD Setup Guide (which covers the GitHub Actions schedule path) — or contact us for a white-glove rollout. This page intentionally keeps things at the decision level.
Section 09 · Output

Where Output Goes

Every routine produces a structured report. Where that report lands is your call — we support every common destination, and the routine config is a one-line swap to switch.

G
Google Chat · recommended default
Plain text format (not Card v2 — easier to read in chat history, paste-friendly, survives forwards). Set up via incoming webhook URL in your Chat space. The same webhook works for all four routines.
S
Slack
Plain text via incoming webhook. Same routines work — just swap the webhook URL. No reformatting, no platform-specific output adapters required.
T
Microsoft Teams
Plain text via Teams connector. Slightly different envelope format (Teams uses MessageCard under the hood), but we keep the inner content plain so it's readable everywhere.
E
Email
Daily digest format. Best for less-chatty teams that prefer scheduled inbox summaries over real-time chat noise. Works with any SMTP provider.
F
File logs
No external dependency. Routines write to local log files on disk. Tail, grep, or pipe into your own observability stack. The most private option.
+
Custom destination
Anywhere a webhook can post — Discord, internal API, PagerDuty, Linear, Jira — works with the same routine. Configure once per destination, reuse across routines.
All output formats are plain text by default — readable, searchable, and copy-pastable. We avoid card-format messages because they break across chat platforms, don't paste cleanly into tickets, and lose meaning when the platform inevitably changes its rendering.
Part 2 · External / CI/CD Setup
Section 10 · CI/CD Setup

Why Run in CI/CD?

Local routines protect the developer. CI/CD protects the team. Once your standards exist as workflow files, every PR gets the same treatment — regardless of who opened it, what laptop they're on, or whether they remembered to install anything.

The shift from local to CI/CD is the shift from advice to enforcement. Local skills produce recommendations a developer can choose to act on. CI/CD checks produce gates — bad code physically cannot merge until the issue is addressed. Both have a place; they cover different failure modes. Local catches issues during development. CI/CD catches issues at the merge boundary, where they would otherwise enter the trunk.
01
Consistency
Every PR gets the same checks. Standards don't depend on who reviewed the code or whether the developer ran the local routine.
02
Blocking
Bad code can't merge. Critical findings — exposed secrets, severe vulnerabilities, policy violations — physically gate the merge button.
03
Auditability
Every run is logged. CI history becomes evidence for compliance audits — when checks ran, what they found, who acknowledged the result.
04
Zero local setup
New hires don't install anything. The standards travel with the repository — clone, push, get reviewed by the team's standard automation.

Section 12 · Comparison

Local vs CI/CD vs Both

A side-by-side at the operational level. If you're still deciding which path to take, this is the table to read carefully — it's the question most engineering leaders ask first.

Aspect Local Setup CI/CD Setup Both (Phased)
Who runs it Individual developer Team-wide automation Both
Cost Your Claude subscription CI minutes (often free tier) Both
Trigger Cron / launchd / on-demand PR push / schedule Both
Output Personal Chat / log Team Chat / PR comments Both channels
Blocking No — advisory only Yes — blocks merge CI blocks; local advises
Setup effort 30 min one-time 1–2 hours one-time 2–3 hours total
Best for Solo work, exploration Team standards, compliance Mature engineering orgs
Privacy Maximum (local only) Standard (CI logs visible) Mixed
Audit trail Personal log Full CI history Both
The honest answer for most teams: start local, add CI/CD when the team grows past five engineers. The two paths are complementary, not competing. CI/CD enforces the floor; local routines raise the ceiling. Mature teams run both because each catches different classes of issue at different points in the workflow.
Section 13 · Phased Rollout

Three Phases, Four Weeks

The rollout pattern most teams converge on. Each phase ships independent value — you can stop after Phase 1 and still get real benefit. The order matters: developers build trust before the gates go up, which means the gates feel like reinforcement rather than imposition.

Phase 1 · Week 1

Individual Adoption

Each developer sets up Claude Code locally. Runs /quick-plan and /code-review on their own work. No team coordination required, no infrastructure changes. The goal is familiarity — developers discover where the skills help and where they don't, on their own terms.

Skills used in Phase 1
quick-plan, quick-work, find-fix, code-standards, code-review
What you ship: zero new infrastructure. Just installed tooling on each developer's machine.
Phase 2 · Weeks 2–3

CI/CD Security Gates

Lead engineer adds the five GitHub Actions workflows. Team standards become enforced automatically — every PR triggers Tier 1 fast gates, PRs to staging trigger Tier 2 code security, infrastructure changes trigger Tier 3, and release branches trigger Tier 4 compliance audits.

Skills wired into CI
Tier 1: secrets-scan, dep-vuln (every PR)
Tier 2: sec-audit, privacy-review, access-audit, audit-log (PR to staging)
Tier 3: cloud-security, infra-harden, network-audit (when infra changes)
Tier 4: hipaa-audit, gdpr-audit, soc2-audit, pci-audit (release branches)
What you ship: 5 workflow files + branch protection rules tying status checks to merge eligibility.
Phase 3 · Week 4 and beyond · Optional

Scheduled Routines

Add cron-based digest routines for teams that want daily and weekly summaries beyond per-PR gating. This is where CI/CD enforcement becomes a continuous engineering pulse — async review, weekly sweeps, and trend visibility for engineering leadership.

Routines added
PR Watch — every 4 hours
Code Quality Pulse — 2–3x per day
Security Sweep — weekly
Full Code Review — weekly
What you ship: cron entries, launchd plists, or hosted scheduler config — no new app code.

By Team Type

Section 15A · Specialized Track

Mobile-Specific Automation

For React Native, Flutter, SwiftUI, or Compose teams. The mobile track follows the same three-phase rollout as the web track — but the skills are platform-aware, and the verification steps include simulator and emulator builds rather than just static checks.

M1
Local mobile skills
mobile-quick-plan runs locally for small mobile tasks — paired with mobile-quick-work for execution. Same flow as the web equivalents, calibrated for mobile context (Expo, native modules, platform parity considerations).
M2
CI mobile skills
mobile-test, mobile-perf, and app-release wire into your release workflow — appropriate for App Store and Play Store submission gates.
M3
Platform verification
iOS simulator builds and Android emulator builds run as part of the CI sequence. Skills produce cross-platform parity reports — flagging behavior differences between iOS and Android before they hit users.
M4
Per-platform routines
Recommended for teams with both iOS and Android engineers — separate scheduled digests per platform so reviewers see only the PRs they own. Reduces noise, sharpens accountability.
Mobile teams typically follow the same three-phase rollout but with mobile-* skill variants. The shape of the rollout is identical; the building blocks are calibrated.
Section 15B · Specialized Track

Compliance Team Routines

For regulated industries — healthcare, finance, government. The compliance track adds scheduled audits aligned to common audit cadences. Output is auditor-ready by default, not a developer log that needs translation later.

Cadence What Runs
Daily secrets-scan deep history scan — catches leaks before they sit overnight in the log stream
Weekly Full Tier 1+2 sweep with HIPAA / GDPR / SOC 2 spot checks on changed files
Monthly Full Tier 1–4 audit with compliance-report evidence package generation
Quarterly incident-plan tabletop exercise prep + access-audit review
Pre-audit (on-demand) Full evidence collection, BAA inventory, control-mapping export — auditor-ready package
Output: the compliance-report skill generates auditor-ready evidence packages. Stored as build artifacts in CI, or as local files in single-developer environments. For more detail on the Compliance Pack itself, see compliance-overview.html.
Section 15C · Reporting

Reporting Cadences

Different audiences need different frequencies. Active developers want real-time feedback; engineering leaders want trends; auditors want quarterly evidence packages. Here's how the cadence ladder typically looks.

Cadence Format Recipients
Real-time PR comment Authors, reviewers
Hourly digest Plain text Chat post Engineering channel
Daily digest Plain text Chat post or email Team leads
Weekly summary Email + Chat Engineering leadership
Monthly report Email + dashboard CTO, Compliance Officer
Quarterly compliance pack PDF / structured docs Auditors, customers
Recommended pairings: per-PR for active development, weekly for trend visibility, monthly for leadership rollups. Most teams over-invest in one cadence and under-invest in another — a balanced ladder beats a comprehensive but rarely-read report every time.
