How to read this catalog

Three rules for using this doc.

This is not philosophy. It's a checklist of mistakes we've already paid for — and the standards that keep us from paying for them again.

Not aspirations. Enforcements.

Each anti-pattern below is something that already happened — at Heaptrace, or at a previous company someone on the team worked at. They are not theoretical. They are documented because they cost time, money, or trust — and because the cost was preventable. If you find yourself reading one and thinking "that's a bit strict," remember: the strictness is what made the cost go away.

The pattern. The fix. The receipt.

Every entry follows the same structure. The Signal — what the anti-pattern looks like at the surface, in code, in chat, in a PR description. Why It's Wrong — the underlying cost, in dollars or trust or time. What We Do Instead — the skill or rule that prevents it, named explicitly. The Consequence — what happens when it shows up anyway, so the standard has teeth. Read all four. Skipping the consequence is how standards rot.

Read once, recognize forever.

The point of this doc is pattern recognition. After reading it once, every code review, every Slack message, every PR description should feel familiar. You'll start spotting these in others' work — and, more usefully, in your own draft before you ship it. The fastest engineers on the team aren't the ones who type the most. They're the ones who recognize an anti-pattern in their own draft and rewrite before anyone else has to.

Planning
5 anti-patterns
❌
01
Planning
Opening the editor without running a plan.
!The Signal

The developer says "let me just take a quick look at the code first" and starts editing. No plan output. No file list. No scope check. Just a cursor blinking in the editor and a vague intent.

?Why It's Wrong

Without a plan, scope grows mid-edit. A 10-line fix becomes a 400-line PR. The reviewer can't tell what was intentional and what was drift. The plan is the cheapest cost-reduction we have — three minutes upfront prevents three hours of rework.

✓What We Do Instead

Every task starts with a plan. /quick-plan for anything under a session. /feature-plan for anything that touches multiple modules or needs design. No exceptions. Even a typo fix gets a 30-second plan — the discipline matters more than the duration.

The Consequence
PR returned at first review with the comment: "Please run /quick-plan and include the output in the PR description." No discussion of the code until the plan exists.
Principle violated: #1 Plan Before Code
Skill that prevents it: /quick-plan /feature-plan
❌
02
Planning
Using /feature-plan for a one-line CSS change.
!The Signal

A heavyweight plan with mockups, risk matrix, and a 14-step breakdown — for a fix that should take five minutes. The developer ran /feature-plan reflexively without checking scope first.

?Why It's Wrong

Token waste, ceremony waste, developer fatigue. /feature-plan is meant for substantial work — when you use it for trivial fixes, you train yourself to skim its output, and then you'll skim it when it actually matters. Devaluing the heavy skill is worse than skipping it.

✓What We Do Instead

/quick-plan for anything that fits in a single session. The scope check at the top of /feature-plan is designed to make you escalate down, not up. If the plan output says "this is an XS task, use /quick-plan instead" — listen to it.

The Consequence
Slower work, frustrated developer, devalued heavy skill. Worse: the next time /feature-plan is needed for an actual feature, the developer skips it because they associate it with overhead.
Principle violated: #1 Plan Before Code
Skill that prevents it: /quick-plan
❌
03
Planning
Using /quick-plan for a 3-day feature.
!The Signal

The plan output says "ETA 3 days, touches 4 modules, needs UI mockups and DB changes — recommend escalating to /feature-plan." The developer ignores the scope check and starts coding anyway.

?Why It's Wrong

/quick-plan can't capture the dependencies, risks, error states, and visual design needed for substantial work. The PR will exceed scope, miss UI states, or land without the migration plan. The scope check exists precisely to catch this.

✓What We Do Instead

When /quick-plan flags XL scope, accept it and run /feature-plan instead. Yes, it costs another 5 minutes. It will save you a day. The escalation isn't a failure — it's the system working.

The Consequence
Feature ships with missing UI states. Design rework adds two days. Reviewer asks "where's the migration plan?" — and there isn't one, because the quick-plan didn't surface it.
Principle violated: #1 Plan Before Code
Skill that prevents it: /feature-plan
❌
04
Planning
Planning, then coding something different from the plan.
!The Signal

The plan says "5 steps, 3 files modified." The PR diff touches 12 files and adds 3 features the plan didn't mention. The plan output sits in the PR description like a fossil from a different project.

?Why It's Wrong

The plan is the contract — with yourself, with the reviewer, with the team. Coding different from the plan defeats the planning step entirely. The reviewer can no longer ask "does the diff match the plan?" because the answer is obviously no, and that question was the cheapest review tool you had.

✓What We Do Instead

If you discover the plan is wrong mid-execution, STOP. Re-run /quick-plan with the new information. Update the PR description with the revised plan. Don't silently expand scope. The five minutes it takes to re-plan is the cheapest part of the work you're already mid-flight on.

The Consequence
Reviewer rejects the PR with: "Diff doesn't match the plan. Re-plan or split." Whatever momentum you had is gone — and the fix is the same five minutes of re-planning you should have done in the first place.
Principle violated: #1 Plan Before Code
Skill that prevents it: /quick-plan /quick-work
❌
05
Planning
Accepting an AI-generated plan without skim-reading.
!The Signal

/quick-plan outputs a plan with an A/B confirmation prompt. The developer hits B within two seconds — no time to have read anything. The terminal scrolls and coding begins.

?Why It's Wrong

The plan may be wrong. The scope may be off. The file list may be incomplete. The AI may have hallucinated a step. Thirty seconds of reading prevents thirty minutes of rework. An unread plan is no better than no plan — worse, because you've now also wasted the plan's tokens.

✓What We Do Instead

Always skim-read the plan before accepting. Check: does the file list match what you expect? Are the steps in a sensible order? Did it miss an obvious risk? If it's wrong, edit it inline or re-run with more context. The verification IS the value.

The Consequence
Half-completed work. Missed acceptance criteria. Reviewer asks "did you read the plan you generated?" — and the honest answer is no.
Principle violated: #5 Verify the AI
Skill that prevents it: /quick-plan

06
AI Usage
Using raw ChatGPT or Cursor for production code on regulated systems.
!The Signal

Developer pastes a prompt into ChatGPT, copies the response into an auth flow, ships it. Or worse: copies generated code from a Cursor side-chat into a payments module without running any of our skills against it.

?Why It's Wrong

Raw AI doesn't enforce our 7 principles. It doesn't carry our project conventions. It doesn't run the scope guards. Output is inconsistent across developers — which means consistency is gone, which means our Principle 4 is gone, which means the whole system breaks down. Skills exist to make every developer ship the same way.

✓What We Do Instead

Skills only for regulated, production-bound, or compliance-sensitive code. Raw AI is fine for rubber-ducking, learning a new library, or non-production prototypes — but the moment code is headed for main, it goes through the skill pipeline.

The Consequence
PR rejected at review. Re-run via the appropriate skill (at minimum /sec-audit if it touches regulated paths). Repeated offenses escalate to a written conversation with the Tech Lead about whether the developer understands the principles.
Principle violated: #2 Skills are the OS
Skill that prevents it: /sec-audit /code-review
❌
07
AI Usage
Trusting AI-generated code without running it.
!The Signal

PR description: "it compiles" or "looks right to me." Not "I ran the tests, clicked the button, checked the DB, verified the response." The developer trusted the skill output and pushed.

?Why It's Wrong

AI hallucinates. The skill minimizes the hallucination rate; verification eliminates the rest. Skipping verification is how hallucinated code reaches production. We have had exactly one incident attributable to this in the past year — and that one cost us a weekend of incident response and a public apology.

✓What We Do Instead

Run the tests. Click the button. Check the database. Every acceptance criterion gets verified before merge — not "I think it works," but "I have evidence it works." Evidence goes in the PR description.

The Consequence
Production incident traceable to skipped verification. We had exactly one. Never again. The next one is no longer treated as an honest mistake — it's a process violation.
Principle violated: #5 Verify the AI
Skill that prevents it: /code-review /test-gen
❌
08
AI Usage
Treating AI-written tests as adequate without checking they fail on the bug.
!The Signal

Bug fix PR. Tests pass on green. Developer didn't run the tests against the un-fixed code. The PR description says "added tests" — but no evidence that the tests would catch the original defect.

?Why It's Wrong

A test that passes both before and after the fix is meaningless — it proves nothing about the bug. AI is especially prone to writing tests that look right but pass against any implementation. The whole point of a regression test is that it would have caught the original bug.

✓What We Do Instead

For bug fixes: verify the test fails on the broken code before applying the fix. Then apply the fix, watch it go green. Include both observations in the PR description: "Test failed at SHA X (pre-fix), passes at SHA Y (post-fix)."

The Consequence
Bug reappears in production despite "fixed and tested." The fastest way to destroy the team's trust in the test suite is to add a regression test that doesn't catch the regression.
Principle violated: #5 Verify the AI
Skill that prevents it: /test-gen /find-fix
❌
09
AI Usage
Using AI as the documentation source for facts we can verify.
!The Signal

"ChatGPT says React 19 has X API..." Or: "Claude told me the Stripe webhook signature is verified using SHA-512..." The developer is treating AI output as a primary source for verifiable facts.

?Why It's Wrong

AI hallucinates docs. It will confidently invent API signatures, deprecated methods, config flags that don't exist, and version numbers that have never shipped. Verifiable facts — API signatures, schemas, syntax, version numbers — should be looked up in primary source, not asked of a language model.

✓What We Do Instead

Use AI to navigate ("find me the docs for X") or to summarize after you've read primary source ("summarize what I just read"). Never use it as the source of truth for API signatures, library behavior, or schemas. Five seconds in the actual docs beats a confident hallucination every time.

The Consequence
Wrong API call ships. Bug surfaces in production. Root-cause analysis traces it to a hallucinated function signature. The fix is documentation discipline, not a better prompt.
Principle violated: #5 Verify the AI
Skill that prevents it: /explain
❌
10
AI Usage
Pasting an AI-generated migration without reading line-by-line.
!The Signal

"It looks like the migration I described." The developer skimmed the SQL, didn't read each statement, didn't trace the rollback path. The migration is in a PR with a one-line description.

?Why It's Wrong

AI-generated SQL has — in real, documented incidents at this firm and others — hallucinated migrations into deleting columns, dropping tables, or running irreversible operations. The model isn't malicious; it's just confidently wrong about a destructive operation. The blast radius of a bad migration is the entire production database.

✓What We Do Instead

Every migration is line-by-line reviewed by the author before submission. Run on local DB first. Run on staging with production-like data second. Verify that rollback is possible. /db-design and /migration-plan exist specifically for this — they catch the destructive patterns the eye misses.

The Consequence
This is the one production incident we've had. Not the AI's fault — the developer's. The migration dropped a column the model "thought" was unused. Recovery cost us four hours of downtime and a credit on three client contracts. The fix isn't a better model. The fix is reading every line.
Principle violated: #5 Verify the AI
Skill that prevents it: /db-design /migration-plan

11
Code & Review
Pushing a PR without running /code-review.
!The Signal

PR opens with no /code-review output in the description. No mention of what the skill said. The developer pushed straight from "code complete" to "ready for review."

?Why It's Wrong

/code-review catches the 80% of issues a human reviewer would catch — but in seconds, not hours. Skipping it wastes the reviewer's time on issues a robot could have flagged, and it lengthens review cycles for every PR across the team. The skill exists to make human review focus on architecture and judgment, not formatting and unused imports.

✓What We Do Instead

Run /code-review on every PR. Include its output — verbatim — in the PR description. The reviewer reads the skill output first, then the diff. The skill is a co-reviewer, not a substitute.

The Consequence
PR returned at first review with: "Please run /code-review and paste the output." This is the most common anti-pattern we see — it happens weekly, and it shouldn't.
Principle violated: #2 Skills are the OS
Skill that prevents it: /code-review
❌
12
Code & Review
Treating /code-review warnings as "I'll fix later."
!The Signal

PR description: "/code-review flagged X but I'll address in a follow-up PR." No follow-up PR is opened. Six months later, X is still there.

?Why It's Wrong

Follow-up PRs are 5x less likely to land than blocking issues fixed in the originating PR. The context is hottest right now — once you context-switch, the cost of returning is high enough that "later" usually becomes "never." Defects compound. Six months of "later" is how a codebase rots.

✓What We Do Instead

Either fix the warning in this PR, or explicitly document why it doesn't apply — with reviewer sign-off and a justification longer than one sentence. "I'll fix later" is not a justification.

The Consequence
Tech debt accrues silently. Six months later, the warning is still there, now buried under newer code that depends on the original mistake. The cost of fixing it has 10x'd.
Principle violated: #7 Standards have teeth
Skill that prevents it: /code-review /tech-debt-audit
❌
13
Code & Review
Refactoring adjacent code in the same PR ("while I'm in here").
!The Signal

PR diff includes 200 lines of unrelated cleanup alongside the actual feature. PR title: "Add feature X". PR description: "Also cleaned up some old helpers while I was in here."

?Why It's Wrong

The reviewer can't separate intentional change from cleanup. PR takes 4x longer to review. Increased blast radius if rolled back — you can't revert the feature without also reverting the cleanup, and vice versa. The temptation is real; the discipline is to resist it.

✓What We Do Instead

Adjacent cleanup goes in a separate PR. One PR = one purpose. If the cleanup is genuinely required for the feature to work, mention it in the plan first, so the reviewer expects it. /pr-strategy teaches the splitting discipline.

The Consequence
PR returned with: "Split into two — the feature change and the cleanup — so each can be reviewed and reverted independently."
Principle violated: #4 Consistency
Skill that prevents it: /code-review /pr-strategy
❌
14
Code & Review
PR description: "fixes bug".
!The Signal

The PR description field has fewer than 20 words. No mention of what changed, why, or how it was verified. The body might be empty entirely. The commit message reads "fix" and the diff is 400 lines.

?Why It's Wrong

Future engineers — including you in three months — can't understand the change. Bisecting a regression becomes guessing. Onboarding a new hire becomes impossible. The PR description is for the team's future self, not for the reviewer today.

✓What We Do Instead

Every PR description includes: what changed, why, how to verify, and the skill outputs (/code-review, /sec-audit, etc.). /smart-commit generates structured PR descriptions automatically — there is no excuse for an empty body.

The Consequence
PR returned at first review with: "Please write a real description. Use /smart-commit if you need help."
Principle violated: #6 Communication is Engineering
Skill that prevents it: /smart-commit
❌
15
Code & Review
Skipping /sec-audit on auth, payments, or PII code.
!The Signal

PR touches an auth flow, a payment processor, or a PII-handling path. No /sec-audit output in the description. Reviewer has to ask: "did you run sec-audit on this?"

?Why It's Wrong

These code paths are where breaches happen. The cost of a missed vulnerability is months of incident response, regulatory exposure, lost client trust, and in regulated industries, real fines. /sec-audit catches the standard OWASP-class issues in under a minute. Skipping it is unforgivable on sensitive paths.

✓What We Do Instead

/sec-audit is MANDATORY on any auth, payments, or PII code. Output goes in the PR description. If the skill flags anything, it gets resolved or explicitly waived (with Tech Lead sign-off) before merge.

The Consequence
PR blocked at CI tier-2 gate (automated) — the pipeline detects touched-sensitive-paths and refuses merge without sec-audit output. Re-submit with the audit attached. This is non-negotiable.
Principle violated: #2 Skills are the OS
Skill that prevents it: /sec-audit

16
Communication
Apologizing to a client before diagnosing the issue.
!The Signal

Client emails: "the service feels slow today." Developer's reply, within two minutes: "So sorry for the issue! We're looking into it right now." No diagnosis yet. No evidence anything is wrong on our side.

?Why It's Wrong

You don't know yet if there IS an issue on your end. The slowness could be the client's local network. Premature accountability trains the client to expect apologies — then to expect blame-acceptance — then to expect compensation. You're setting the bar for every future incident, and you're setting it before you even know what happened.

✓What We Do Instead

"Looking into it; will report back within 30 minutes." Accountability comes after diagnosis. If it turns out to be our issue, then we own it clearly. If it turns out not to be, we don't — and the client respects the discipline.

The Consequence
Long-term client relationship erodes. You become the team that's always apologizing — and apologetic teams don't get the next contract. Clients hire confident engineering, not contrite engineering.
Principle violated: #6 Communication is Engineering
Skill that prevents it: /message-craft
❌
17
Communication
Sending a client email without /message-craft review.
!The Signal

Direct client-facing email goes out with hedging language, premature apology, a buried headline, or a permission-seeking close. The developer wrote it and sent it — no /message-craft pass. "It was just a quick reply."

?Why It's Wrong

Client trust is more correlated with comms quality than with actual MTTR. We have data on this. A great fix delivered with a bad email is rated lower than a slow fix delivered with a great email. The communication is part of the deliverable.

✓What We Do Instead

/message-craft on every client-facing message — even "quick replies." Especially quick replies, because that's where the bad habits show up. The skill takes 90 seconds. The bad email costs 90 days of trust.

The Consequence
Account-management escalation. Trust score drops. The Account Lead pulls client comms out of engineering's hands — which is exactly the autonomy we don't want to lose.
Principle violated: #6 Communication is Engineering
Skill that prevents it: /message-craft
❌
18
Communication
Burying the headline in long context-dumps.
!The Signal

First three paragraphs of an email describe what the developer has been thinking. The actual update — "the deploy is delayed by 24 hours" — is sentence 14, after three paragraphs of context, troubleshooting notes, and ideas-in-progress.

?Why It's Wrong

Most readers stop at line two. Executives stop at line one. Your headline must carry the message even if they read nothing else. Buried headlines aren't thorough — they're disrespectful of the reader's time, and they hide the very information you're being paid to deliver.

✓What We Do Instead

Headline first. Then context, if asked. BLUF — Bottom Line Up Front. /message-craft enforces this structure: the first line is the takeaway, everything else is support. If your reader stops at line one, they still got the message.

The Consequence
Client misses the update entirely. Chases you the next day asking for status that was already in the email — because they only read the first paragraph, and the first paragraph wasn't the headline.
Principle violated: #6 Communication is Engineering
Skill that prevents it: /message-craft
❌
19
Communication
Using hedging language in client comms.
!The Signal

"I think we can probably have a fix by tomorrow." Or: "This might be related to..." Or: "It's possible that..." Every sentence hedged with "I think," "maybe," "probably," "kind of."

?Why It's Wrong

Hedging undermines confidence. It reads as junior. Clients trust leaders who name dates and stand by them. There is a difference between "I don't know yet, I'll have an answer by 5pm" (confident uncertainty) and "I think probably maybe tomorrow" (hedged uncertainty). One is leadership. The other is not.

✓What We Do Instead

"Engineering is on it. Deploy targeted for tomorrow EOD." Set the timeline. Own it. If you're not sure, name the next checkpoint: "We'll have a confirmed ETA by 3pm." /message-craft flags hedging language automatically.

The Consequence
Client follows up the next day asking "is it actually fixed?" — because you never said it would be. You said you thought it might be. Those are different sentences.
Principle violated: #6 Communication is Engineering
Skill that prevents it: /message-craft
❌
20
Communication
Closing client emails with "let me know if that works."
!The Signal

Email ends with: "Let me know if that works for you!" Or: "Does this seem okay?" Or: "Open to your thoughts." A permission-seeking question to a client.

?Why It's Wrong

Clients want decisions, not options. They hired us to know the answer. Permission-seeking reads as you don't know either. It transfers the decision weight back to the client — which is exactly the work they're paying us to do.

✓What We Do Instead

State the plan. Offer to discuss only if needed: "Will proceed Tuesday unless you flag a concern by EOD Monday." Default to action. The client can always object — and most won't, because the plan was reasonable, and you gave them an explicit out.

The Consequence
Client reads it as you're not sure. Trust erodes. Worse: the decision sits in the client's inbox for three days, blocking work that could have proceeded.
Principle violated: #6 Communication is Engineering
Skill that prevents it: /message-craft

21
Security
Skipping /dep-vuln when adding a new dependency.
!The Signal

package.json (or requirements.txt, Gemfile, go.mod) gains a new entry. PR has no /dep-vuln output. The dependency is some lesser-known package the developer found on Stack Overflow.

?Why It's Wrong

Supply chain attacks happen through new dependencies — typosquatted packages, compromised maintainer accounts, malicious version bumps. A 30-second scan with /dep-vuln catches the known-bad ones. Skipping it is how a backdoor ends up in your bundle.

✓What We Do Instead

/dep-vuln on every PR that touches dependency files. Output goes in the PR description. If the skill flags anything, resolve it or pick an alternative package — and document why if you're proceeding anyway.

The Consequence
CI tier-2 gate blocks merge automatically. You don't get to override it without Tech Lead written exemption. The pipeline is the enforcement layer precisely because manual gates fail in a rush.
Principle violated: #2 Skills are the OS
Skill that prevents it: /dep-vuln
❌
22
Security
Bypassing scope guards because "the skill got it wrong."
!The Signal

/quick-plan flags XL scope, recommend /feature-plan. Developer says "that should still be doable in a session" and starts coding. No escalation. No Tech Lead conversation.

?Why It's Wrong

The scope check is calibrated. We tuned it on real PRs. When it flags XL, escalating to /feature-plan is cheaper than mid-flight scope correction. Bypassing the guard because you think it's wrong is the same argument every junior makes — and 90% of the time, the guard is right.

✓What We Do Instead

Trust the scope check by default. If you genuinely disagree, document why in writing and have a Tech Lead sign off before proceeding. The friction is the point — it forces the conversation that prevents the rescue mission later.

The Consequence
Mid-week scope rescue. Missed deadline. Tech Lead pulled in to triage. Whatever speed you thought you were gaining by skipping the escalation is now lost three times over.
Principle violated: #7 Standards have teeth
Skill that prevents it: /quick-plan /feature-plan
❌
23
Security
Shipping a skill warning unresolved without explicit acceptance.
!The Signal

/sec-audit flagged a moderate-severity risk. Developer left it in the PR description with no explanation. The reviewer assumes it must be intentional. The Tech Lead assumes the reviewer addressed it. Everyone assumes someone else handled it.

?Why It's Wrong

Skill warnings are real. Ignoring them silently is worse than not running the skill — because now there's a paper trail showing the team saw the warning and shipped anyway. That's the kind of artifact that turns a minor incident into a major liability issue.

✓What We Do Instead

Either fix the warning, OR explicitly document acceptance with (a) a reason it doesn't apply in this context, and (b) Tech Lead sign-off. Both in writing, both in the PR description, both before merge.

The Consequence
PR blocked at review. Explicit written acceptance required before merge. If you ignore this pattern repeatedly, your PRs start requiring Tech Lead approval on every change — your autonomy is what's at stake.
Principle violated: #7 Standards have teeth
Skill that prevents it: /sec-audit (any skill)
❌
24
Security
Hardcoding secrets, even "just for local testing."
!The Signal

An API key, password, JWT secret, or token appears in a code file. Comment next to it: "// TODO: move to env before merging" or "// just for local testing". The TODO never gets done. The commit lands.

?Why It's Wrong

Git history is forever. "Just for local" leaks through shared dev branches, CI logs, screen-shares with clients, and accidental pushes to public forks. Once a secret is in git history, rotation is the only remedy — and rotation is expensive when third-party systems are involved.

✓What We Do Instead

.env files (in .gitignore) for local. Proper secrets management (Vault, AWS Secrets Manager) for staging and production. /secrets-scan runs in CI and catches the obvious patterns; /secrets-manage is the design discipline for getting it right from day one.

The Consequence
Secret rotation required. Security incident report filed. Audit trail of the leak preserved for compliance review. If the secret was a client's API key, add an apologetic client email to the cost.
Principle violated: #5 Verify the AI
Skill that prevents it: /secrets-scan /secrets-manage

Process & Discipline
4 anti-patterns
❌
25
Process
"Let it slide just this once" on a standard.
!The Signal

A reviewer waives a missing /code-review output "because it's a small change." Or a Tech Lead approves a missing /sec-audit "because we're rushing to ship before the demo."

?Why It's Wrong

Exceptions compound. The next exception is harder to deny because the precedent exists. Within three months, no one runs the skill because "it's fine, we let it slide last time." Standards are eroded one slide at a time — and the slide is always plausibly small in the moment.

✓What We Do Instead

Standards are bendable in writing only. Tech Lead documents the exemption — what was waived, why, and an expiration date. No verbal exceptions. /pr-strategy describes the formal exemption process.

The Consequence
The standard becomes a suggestion. Within three months, no one runs the skill. This is the gateway anti-pattern — it's the one that lets every other anti-pattern in.
Principle violated: #7 Standards have teeth
Skill that prevents it: /pr-strategy
❌
26
Process
Senior rewriting a junior PR to "fix the structure."
!The Signal

Senior takes over the PR. Force-pushes the fix. Junior moves on. The commit log shows the senior as author. The junior never sees the rewrite, never learns what was wrong, never improves.

?Why It's Wrong

Junior doesn't learn. The pattern repeats next sprint. The senior becomes the bottleneck — every junior PR routes back to them for rewrites. And the junior plateaus, because the feedback loop that would have made them better was short-circuited by helpfulness.

✓What We Do Instead

Bounce the PR back with specific, actionable feedback. Junior re-submits. Slower this PR, faster the next ten. /onboard-dev and /code-review are the tools — both pair a skill output with human feedback so the junior can map their mistake to a named pattern.

The Consequence
Junior plateaus. Senior burns out. Six months later, you're hiring another senior because the current one can't take any more rewrite duty. The short-cut became the bottleneck.
Principle violated: #3 Skills flatten the curve
Skill that prevents it: /code-review /onboard-dev
❌
27
Process
Treating SE I/II PRs as rubber-stamp reviews.
!The Signal

Reviewer approves the PR within 30 seconds with: "looks good 👍". No substantive comments. No questions. Just a quick LGTM and a merge.

?Why It's Wrong

Even with skills, code review is where craft transfers. Skipping it stunts growth, ships defects, and trains the junior to expect their PRs aren't worth reading. The skill handles surface issues. Humans do architectural review. One does not replace the other.

✓What We Do Instead

Every PR — from any level — gets real review. /code-review handles the first pass; the human's job is to ask the architectural questions the skill can't. "Why this approach over X?" "What happens when Y fails?" "Did we consider Z?" Those questions are how the junior learns to ask them of themselves.

The Consequence
Defects ship. Juniors stall. Three months in, the SE I is still writing SE I code, because nobody's pushed them to the next level. That's the reviewer's fault, not the junior's.
Principle violated: #4 Consistency requires review
Skill that prevents it: /code-review
❌
28
Process
Making exceptions to skills without written exemption.
!The Signal

Slack: "Just for this PR, we're skipping /sec-audit because we're in a rush for the demo." No written exemption. No expiration. No conversation about what happens to the skipped audit after the demo.

?Why It's Wrong

Skills exist precisely for high-pressure moments. Skipping them under pressure is exactly when they catch the most — because pressure is when developers cut corners. Verbal exceptions in Slack don't survive — three weeks later, no one remembers the context, and the skipped audit is still skipped.

✓What We Do Instead

No verbal exceptions. If a skill must be skipped, written Tech Lead exemption with: (a) the explicit reason, (b) the date by which the skill will be run retroactively, and (c) the issue tracker ticket holding the follow-up. Three conditions, all in writing.

The Consequence
CI gate fails. Merge blocked. The rush turns into a longer delay than the skill would have caused — because now you're in a Slack thread debating whether to break process, instead of in the IDE running the skill.
Principle violated: #7 Standards have teeth
Skill that prevents it: (context-specific)

Role-Specific
SE III/IV hoarding architectural decisions in their head.
!The Signal

Senior makes the architectural call in a DM. Or in a face-to-face conversation. Or in their own head, without telling anyone. Six months later, no one remembers why the system is shaped that way — including the senior.

?Why It's Wrong

Patterns become tribal knowledge. New hires re-learn the same lessons because the lessons aren't written down. The senior becomes irreplaceable in a bad way — if they leave, the architectural intent goes with them. The org pays for the debate twice: once when it happened, once when it gets re-debated.

✓What We Do Instead

/decision-doc for every architectural choice. Stored, searchable, citable. Every "we chose X over Y because Z" gets a document. No exceptions for senior engineers — the more senior, the more important the documentation, because more downstream decisions depend on it.

The Consequence
Same architectural decision re-debated three months later. The org pays for the debate twice. Worse: a new hire builds on the wrong assumption because they couldn't find the original rationale.
Principle violated: #7 Standards have teeth
Skill that prevents it: /decision-doc
❌
30
Role-Specific
SE IV operating at SE III level (reviewing PRs instead of designing systems).
!The Signal

SE IV's calendar is 80% PR review, 5% architecture, 15% meetings. No blocks of time for system design, hiring, or handbook work. Asked about the org-wide migration plan, they say "haven't gotten to it yet."

?Why It's Wrong

SE IV time is for systems, hiring, mentorship, and the handbook itself. Routine PR review is SE II/III work. When an SE IV operates at the SE III level, the SE III level is over-saturated, the SE IV's actual job goes undone, and the whole leveling structure collapses downward.

✓What We Do Instead

SE IV reviews PRs only when architectural concerns are at stake — cross-system designs, security-critical paths, new patterns being introduced. Routine review is delegated. /system-design and /arch-review are how SE IV time is spent — at the level the level demands.

The Consequence
Org-wide architecture decisions don't happen. Systems drift. The next migration is bigger because the small migrations didn't get planned. Compensation for SE IV becomes hard to justify because they're producing SE III output.
Principle violated: #4 Consistency
Skill that prevents it: /system-design /arch-review
Watch for these

The five most common.

Of the thirty patterns above, these five account for roughly 80% of what we see in PR review. If a team is going to enforce only five, enforce these.

⚠
High-frequency anti-patterns
Based on internal PR review data, last 12 months
  1. #11 Pushing without /code-review
    Most common by far. Happens at least weekly across the team. The "I'll add it before merge" promise rarely materializes. Easy to enforce mechanically via PR template and CI gate — but the discipline has to come from the developer first.
  2. #13 Refactoring adjacent code in the same PR
    Subtle, requires PR discipline. Every developer has the urge — "while I'm in here, I'll clean this up." Resist. The split into two PRs costs five minutes and saves the reviewer an hour.
  3. #16 Apologizing to clients before diagnosis
    Instinctive — the hardest to unlearn. The reflex to apologize is empathetic and human, but in client communication, it's also untrained accountability. Replace reflex apology with "Looking into it; report back within 30 min."
  4. #22 Bypassing scope guards
    Developer pride at play. "I can do this in a session, the skill is wrong." Sometimes the skill is wrong — about 10% of the time. The other 90% is the developer underestimating, and the cost of the underestimate is the rescue mission.
  5. #25 "Let it slide just this once"
    The gateway to all other anti-patterns. Reviewers and Tech Leads — this one is on you. Every slide compounds. Every exception is a precedent. The friction is the feature.
The 80/20 rule for this doc: If your team isn't ready to enforce all thirty, start with these five. They'll cover the majority of the damage — and they'll train the muscle for the rest.
