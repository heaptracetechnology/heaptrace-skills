Not a list. A decision system.

We have 101 skills. You don't need to know all of them — you need to know which one fits the moment you're in. This playbook is organized around the moments, not the skills. Start at "what are you trying to do?" and follow the tree. The catalogue at the bottom is for recall, not for learning.

Pairings beat single skills.

The biggest productivity gains come from chains: /find-fix → /quick-plan → /quick-work → /smart-commit. A bug becomes a fix becomes a commit in one workflow. Section 5 maps the chains. Memorize the chains, not the individual skills — the chains are what change a Monday morning.

Section 4 · Daily Workflows

Ten chains you'll run every week.

These are the sequences. Memorize them — or bookmark this section. The chains are where the real productivity lives; single skills are how engineers without skills work.

Small Task (Bug Fix or Tweak)
Workflow 01
/find-fix → /quick-plan → /quick-work → /code-review → /smart-commit
Why Finds the root cause, plans the scoped fix, executes, reviews, commits. The whole flow in under 30 minutes for an S-sized task. Skip /find-fix if you already know the root cause; never skip /quick-plan.
New Feature (One Session, Multi-File)
Workflow 02
/feature-plan → /feature-work → /test-gen → /sec-audit → /code-review → /smart-commit → /release-notes
Why Full ceremony for substantive work — plan, implement, test, audit, review, commit, release notes. /sec-audit is required if the feature touches auth, payments, or PII. /release-notes runs only when the feature ships.
Mobile Task (iOS or Android)
Workflow 03
/mobile-quick-plan → /mobile-quick-work → /code-review → /smart-commit
Why Mobile-aware planner and implementer pair. Platform field, platform quirks, native verify commands (xcodebuild, Gradle). Use /mobile-feature instead of /mobile-quick-plan when the work spans both iOS and Android navigation, state, and UI.
Compliance / Regulated Code
Workflow 04
/feature-plan → /feature-work → /sec-audit → /hipaa-audit → /compliance-report → /code-review → /smart-commit
Why Regulated work needs framework-specific audits inline, not at the end. Swap /hipaa-audit for /gdpr-audit, /soc2-audit, or /pci-audit depending on the regime. /compliance-report aggregates findings into the evidence packet clients ask for.
Architecture Decision
Workflow 05
/arch-review → /decision-doc → /scalability-review → /system-design → /integration-plan
Why SE III/IV territory. Decision flows from review → ADR → (if green-field) system design → integration plan. /scalability-review fires when the decision affects load characteristics. /decision-doc is the artifact that survives the conversation — write it before you implement.
Incident Response (Active Incident)
Workflow 06
/incident-response → /find-fix → /quick-plan → /quick-work → /message-craft → /incident-plan
Why Active incident mode: triage → diagnose → fix → communicate → post-mortem. /message-craft is mandatory the moment a client is affected — never drop client comms. /incident-plan writes the post-mortem within 48 hours, not "later."
Pre-PR Quality Gates
Workflow 07
/code-review → /sec-audit → /test-coverage → /smart-commit
Why The standard pre-merge gate. Every PR runs this minimum. Security audit drops in for auth, payments, PII, or anything that touches a regulated boundary. Reference Principle 7: standards have teeth.
Sprint Planning
Workflow 08
/sprint-plan → /feature-prioritize → /user-story → /scope-negotiate → /metrics-define
Why PM + Tech Lead collaboration. Plan, prioritize, write stories, negotiate scope, define success metrics. /scope-negotiate runs when the ask exceeds the capacity — and it always does. Write the metrics before the sprint starts, not after.
Onboarding a New Engineer
Workflow 09
/onboard-dev → /code-standards → /pr-strategy → /explain
Why New hire's first month. Tech Lead invokes onboarding flow; new engineer learns standards through real-context explanations rather than abstract docs. /explain runs continuously — every time the new engineer hits unfamiliar code.
Cloud Infrastructure Change
Workflow 10
/cost-estimate → /system-design → /terraform-module → /iam-policy → /ci-cd-pipeline → /monitoring-setup → /disaster-recovery
Why Cloud Engineer Pack flow for any new AWS resource or significant infra change. /cost-estimate first — never ship infra without a cost number. /disaster-recovery runs as a checkpoint for stateful resources (RDS, S3 with critical data, ECS clusters running paid workloads).
Section 5 · Natural Pairings

Skills that always travel together.

If you've invoked the first skill, you'll almost always need the second. These pairs encode the muscle memory of skilled developers.

First Skill Naturally Pairs With Why
/quick-plan /quick-work Same plan, immediate implementation. Don't context-switch between them.
/feature-plan /feature-work Same plan, full implementation cycle. The plan is wasted if it doesn't drive the work.
/find-fix /quick-plan → /quick-work Diagnose root cause → plan the scoped fix → implement. Skipping the plan is how 15-minute bugs become 2-hour debugging sessions.
/mobile-quick-plan /mobile-quick-work Platform-aware plan, platform-aware implementation. Native verify commands carry between them.
/sec-audit /hipaa-audit · /gdpr-audit · /soc2-audit · /pci-audit General security pass + framework-specific findings. Both are required for regulated code; one is not a substitute for the other.
/code-review /test-gen Review surfaced untested code paths. Generate the tests in the same session — don't add a backlog item.
/arch-review /decision-doc The review surfaces a decision. Document it as an ADR while the reasoning is still fresh, not later.
/feature-plan /api-design + /db-design Substantive features need API contract and schema before code. Plan → contract → schema → implement is the order.
/incident-response /message-craft Live incident, client comms are mandatory. The fix and the message ship in parallel, not in series.
/migration-plan /scalability-review Migrations validate future capacity. Always check that the destination can handle three years of growth, not just today's load.
/sprint-plan /user-story Sprint planning needs decomposed stories with acceptance criteria. Stories before estimates.
/system-design /cost-estimate Every system design carries a cost. Estimate before you build; clients ask for the number.
/api-design /contract-test Contract defined → contract tested. Pact or schema validation tests get written with the contract, not after the consumer breaks.
/wireframe /user-flow → /design-system Wireframes need flow context, then resolve into design system primitives. Don't ship one-off components.
/dep-vuln /sec-audit Dependency CVEs surface vulnerable libraries. Security audit verifies whether the vulnerable paths are actually exposed.
/smart-commit (always last) Final step before push. Writes the conventional commit message, runs the pre-commit hooks, surfaces what's actually changing.

Section 7 · Honest Trade-offs

Not every skill fits every task.

A heavy skill on a light task is wasted tokens, wasted time, and a signal that you don't understand what you're doing. Skills are tools. Pick the lightest one that fits.

  • /feature-plan is overkill for a one-line CSS change or a typo fix. Use /quick-plan — or for genuinely trivial edits, just plan inline in two bullets and ship.
  • /sec-audit is overkill for a copy change, a UI tweak, or a config update with no auth surface. It is mandatory for anything touching auth, payments, PII, or a public API surface. The line is: does this change who can do what, or who can see what?
  • /system-design is for new systems, not feature additions to existing ones. If your team has been running the service for two years, you don't need a system-design doc for the next feature — you need /feature-plan.
  • /cost-estimate matters at the cloud-infra level and at commitment time. Don't run it for every feature; do run it for any new AWS resource, any change to autoscaling bounds, or any third-party integration with usage-based pricing.
  • /load-test applies to traffic-sensitive endpoints — authentication, checkout, search, anything that fans out to multiple downstream services. Don't load-test every CRUD operation. Do load-test before scaling events (launch, marketing push, enterprise rollout).
  • /decision-doc is for decisions that will outlast the person making them. If the choice is reversible in an hour, you don't need an ADR. If it's reversible in a week, you might. If it's not reversible at all, you absolutely need one.
  • /migration-plan is for cutovers — database, framework, service, vendor. Not for renaming a file or refactoring within a module. The test is: is anyone outside your team affected by the change? Then yes, you need a plan.
  • /onboard-dev is for human onboarding, not project setup. Pair it with /code-standards at the same time so the new engineer learns standards through real code, not abstract docs.
  • /competitor-analysis belongs to product, not engineering. If an engineer is running it, the team probably has a scoping problem that should be escalated to the PM or Tech Lead — don't paper over it.
  • /sprint-plan is for the team rhythm, not for picking up an ad-hoc ticket. If you're just grabbing a card from the backlog, run /quick-plan and start.
Token discipline matters too. Right-sized planning is the entire point of Principle 1 — Plan Before Code, Even for Small Tasks. "Even for small tasks" doesn't mean "every task gets a 1,500-line spec." It means every task gets a plan proportional to its size. /quick-plan for small. /feature-plan for substantive. /system-design for green-field. Pick the lightest skill that fits, and ship. See Principle 1 for the right-sizing rules.
Section 8 · The Anti-Skill

Raw AI is not a skill.

The biggest threat to our quality bar isn't an engineer making a mistake — it's an engineer bypassing skills because "the autocomplete already wrote it." Reference Principle 2.

Don't use raw ChatGPT, raw Cursor autocomplete, or raw Copilot for production code on regulated systems.

Raw AI doesn't enforce our 7 principles. Skills do. Raw AI doesn't generate consistent output across developers. Skills do. Raw AI doesn't carry our project conventions, our security gates, our compliance posture, or our communication standards. Skills do all of that, by design.

The output of raw AI is a function of the prompt. The output of a skill is a function of the prompt plus our standards plus our project context plus our role-aware checks. The variance between those two is the entire reason we exist as a team.

Raw AI is fine for: rubber-ducking an approach, search-engine-substitute questions ("what's the syntax for X"), prototyping non-production code, drafting an email or doc that you'll edit by hand. None of that ships.

Anything that ships, ships via a skill. If you find yourself reaching for raw Cursor autocomplete on production code, stop. Open a skill. Run it. Ship the skill's output. That's the only workflow on regulated code.
