The 7 Principles
of AI-First Engineering

The non-negotiables. These are the only opinions our team has to share. Everything else is up for discussion.

Back to Handbook Skills Playbook
Why these seven

Not aspirations. Operating instructions.

A handbook is only worth the behavior it changes. Read these once, then go work. Refer back when something feels off.

Not aspirations. Operating instructions.

A principle that doesn't change a developer's daily behavior isn't a principle — it's marketing. These seven change behavior. Each one names what we do, what we don't do, and what the consequence is when the principle is broken. If you finish reading and your Monday morning looks identical to last Monday, we've failed to write a principle.

Two readers in mind.

We wrote these for two readers at the same time: a Software Engineer I joining their first day at Heaptrace, and a client evaluating whether to extend their engagement. The principles read the same to both. If they didn't, we wouldn't believe them. Internal honesty and external clarity are the same standard here, written in the same words.

Read once. Reference forever.

Each principle is intentionally short. No essay. No nuance dump. The team should be able to recite all seven from memory. The clarification belongs in the supporting docs — How We Work by Role, the Skills Playbook, Anti-Patterns, Quality Gates. This page is the spine. The rest is the body.

The seven

One opinion per principle.

No "it depends." No hedging. Each principle below is what we do, full stop — with the anti-pattern and the consequence named in plain language.

01
Principle One

Plan before code, even for small tasks. The plan determines the skill. The skill determines the rigor.

+ What this looks like in practice
  • Small task (under one day) — run /quick-plan, then /quick-work. Two skills, four minutes of upfront thinking, scope locked.
  • Larger feature — run /feature-plan, then /feature-work. Full mockups, data flow, risk register, acceptance criteria — all before the first line of code.
  • Bug — run /find-fix first to diagnose, then /quick-plan or /feature-plan based on the scope the diagnosis reveals.
  • The plan is reviewed by the developer (and a teammate, if multi-day) in 30 seconds before coding starts. Plans that take longer than 30 seconds to read are rewritten.
− What we don't do
  • Open an editor and start typing on anything bigger than a typo or a copy change.
  • Use /feature-plan for one-line changes — that's token waste and ceremony waste, and it makes the team cynical about the skill.
  • Use /quick-plan for work that spans multiple sessions or multiple developers. If two people are touching it, it's a feature.
  • Skip planning because "I already know what to do." That's the moment to write it down — fast plans for known work are still plans.
! Consequence when broken
PRs without an upfront plan exceed scope roughly eight out of ten times in our internal review data. The pattern is identical every time: the developer starts coding, hits a fork, picks the more interesting path, and ships work that wasn't asked for. Plans aren't optional — they're the cheapest cost-reduction mechanism we have.

02
Principle Two

The skill is the operating system, not the suggestion. If a skill exists for the task, the skill is run — not "consulted."

+ What this looks like in practice
  • Every PR that touches auth, payments, or PII runs /sec-audit before review — without exception, regardless of how trivial the change looks.
  • Every PR that adds dependencies runs /dep-vuln. New libraries don't enter the codebase without a vulnerability scan in the PR.
  • Every PR description includes the output (or a one-line summary) of any skill that was run, so the reviewer can verify it was actually run and not skipped.
  • New patterns we discover get codified as skills, not as Slack messages. If something is worth telling the team once, it's worth turning into a skill that runs forever.
− What we don't do
  • Use raw ChatGPT or Cursor autocomplete on regulated code (auth, billing, PII, healthcare data, financial calculations).
  • "Glance at" a skill's output and ignore it. Either you resolve every finding or you explicitly accept it in writing in the PR description.
  • Ship a skill's warning unaddressed. Warnings either get fixed or get a documented exemption signed by a Tech Lead.
  • Add new conventions verbally in standup. A convention that isn't a skill rule doesn't exist on this team.
! Consequence when broken
Skills are reviewed by three or more domain experts before they ship. Skipping the skill means trusting one developer's judgment over the codified expertise of three. That trade is rarely worth it — and the times it appears to be worth it are the times we've shipped the bug.

03
Principle Three

A Software Engineer I with skills outranks a Software Engineer III without them. Skills flatten the experience curve.

+ What this looks like in practice
  • SE I developers receive senior-level scaffolding from /feature-plan, /quick-plan, /code-review, and /sec-audit — the same scaffolding a SE IV would use, no watered-down version.
  • A SE I PR backed by skill outputs reads at a SE III level in code review. We've measured this against blinded reviewer scoring.
  • Tech Leads spend their time on judgment calls and architecture, not catching basic mistakes — the skills catch those before the PR opens.
  • We promote on judgment, decisions, and ownership — not on syntactic ability. The skills cover syntax; we hire and grow people for taste.
− What we don't do
  • Assume seniority means a developer doesn't need skills. Often it's the opposite: seniors miss domain-specific anti-patterns that the skill was specifically built to catch.
  • Romanticize "10x engineers" who refuse tooling. On this team that's not a 10x engineer; that's a 0.5x liability dressed up as folklore.
  • Block SE I developers from doing work that skills would let them do. If the skill makes the work safe at SE I, we let SE I do the work — and we don't gate it on years of experience.
! Consequence when broken
We've measured a 3x velocity gap between teams that adopted this principle and teams that resisted it. The teams that resisted ended up with their seniors gatekeeping work and their juniors stuck on tickets that should have been theirs. Skills aren't a crutch — they're the curriculum.

04
Principle Four

Output consistency is the real metric. Same input, same quality output — regardless of which developer is at the keyboard.

+ What this looks like in practice
  • We measure: PR review iterations, defect rate, audit pass rate, time-to-merge variance. Four numbers, reviewed monthly.
  • Five developers given the same ticket should produce indistinguishable PRs — in structure, in rigor, in test coverage — even if implementation details vary. Cross-developer consistency is reviewed quarterly.
  • When variance shows up between developers, we fix the skill (or write a new one), not the developer. The developer is the symptom; the gap in the skill is the cause.
  • Client deliverables follow the same standard PR structure. The PR is the same shape whether it's internal or going to a Fortune 500 client.
− What we don't do
  • Reward developers who ship 10x more code. We've watched that culture eat teams alive — and the cleanup work always falls on the people who weren't being rewarded.
  • Penalize developers who ship 0.5x as much code but with zero defects and clear plans. That's frequently the highest-leverage developer on the team.
  • Use AI adoption as a metric ("did Sarah use Claude Code today?"). That's input, not output. We measure the result, not the tool.
  • Run "lines of code" or "commits per day" reports. We don't, and we won't.
! Consequence when broken
Inconsistency in PRs is the single best predictor of post-merge bugs in our internal data — better than complexity, better than file count, better than developer tenure. Teams with low variance ship cleaner. Period. The metrics we use are the metrics that predicted the outcome.

05
Principle Five

Verify before you trust the AI. The model hallucinates. Skills minimize it. Verification eliminates the rest. Both are non-negotiable.

+ What this looks like in practice
  • Every skill output includes acceptance criteria — and we verify those criteria before merging. The skill is not the proof; the verification is the proof.
  • When a skill says "I checked X," we sample-check it on the first three PRs by a new developer, until we trust the pattern in their hands.
  • Hallucinations are reported back into the skill (issue or PR) so they get fixed at the source. A hallucination caught and not reported is a hallucination that will catch the next developer.
  • We run the tests. We click the button. We check the database. We don't assume.
− What we don't do
  • Merge AI-generated code without running it. Ever. No exceptions, including "it's just a config change."
  • Trust an AI-written summary of "what changed" — read the diff yourself. The summary is a starting point, not the truth.
  • Accept AI-written tests at face value. The test must actually fail when the bug is present. A passing test that doesn't fail under failure conditions is theatre, not coverage.
  • Use AI as a fact source for things we can verify — docs, APIs, schemas, library versions. Look it up. It's faster than fixing the wrong code.
! Consequence when broken
The one production incident attributable to AI in the past twelve months was a missed verification step on a generated database migration. It wasn't the model's fault. It was the developer's. Verification is the developer's job, every time, and "I read it and it looked right" is not verification.

06
Principle Six

Communication is engineering. The PR description, the Slack update, the client email — these are engineering, and they're held to the same standard.

+ What this looks like in practice
  • Every PR description includes: what changed, why, how to verify, any skill outputs. Four sections, every time. No prose paragraphs.
  • Every client-facing message goes through /message-craft on the first draft. The skill catches premature apology, hedging, and scope creep before it leaves your outbox.
  • Incident updates follow NTSB form: facts first, no premature blame, named recovery owner, time-bound next update. If the next update isn't scheduled, the message isn't sent.
  • Status updates use the standard format: Done · In Progress · Blockers · Next. Four labels. No paragraphs. Everyone reads it the same way.
− What we don't do
  • Write "fixes bug" as a PR description. That gets bounced at first review — not rewritten by the reviewer.
  • Send a client email without a 30-second review for premature apology, hedging, or scope creep in commitments. "I'll get back to you with X by Friday" is a contract; treat it like one.
  • Over-apologize, accept blame before diagnosis, or hedge with "I think" in incident communications. Each of those is a credibility leak that compounds over months.
  • Write client-facing prose in casual register. The standard is calm, precise, factual — the same register as a flight crew reading an emergency checklist.
! Consequence when broken
Client trust correlates more with incident communication quality than with actual time-to-recovery — we have the survey data to prove it. Same outage, two different messages, completely different client perceptions. Communication isn't downstream of engineering; it's part of engineering, and the message that ships is part of the deliverable.

07
Principle Seven

Standards have teeth, or they have none. A standard without a consequence is a suggestion. Suggestions don't change behavior. We enforce.

+ What this looks like in practice
  • PRs without /code-review output are rejected at first review — re-submit after running it. The reviewer doesn't run the skill on your behalf.
  • PRs missing the standard description sections are sent back, not rewritten by the reviewer. The author writes the description.
  • Skill outputs that warned and were ignored block merge until explicitly accepted in writing in the PR description, with the reason.
  • New team members who consistently bypass skills get a 1:1 with their Tech Lead within 30 days — not punitive, but firm. We name the pattern early so it doesn't become a habit.
− What we don't do
  • "Let it slide just this once." That phrase is how every broken standard starts. The exception becomes the precedent within two weeks.
  • Rewrite junior PRs to fix structure. They don't learn that way; they learn from bouncing. Reviewing is teaching, not cleaning up.
  • Tolerate process exceptions without an explicit, written exemption from a Tech Lead — and exemptions are logged, not whispered.
  • Apologize for the standards in client conversations. The standards are why the client is hiring us. Don't soften them in the room.
! Consequence when broken
The team that lets standards slide has standards that don't exist. We've watched this play out — at our own clients and in our own past. Once an exception is granted casually, the next request becomes harder to deny. The third one is impossible. Standards are protected by being unbendable; the moment they bend, they break.
