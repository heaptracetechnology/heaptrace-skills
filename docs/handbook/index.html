Why this handbook exists

Variance is the failure mode.

Most engineering organizations adopt AI by handing every developer a license and hoping for the best. We don't. The handbook below is the reason.

AI is not a productivity multiplier unless you remove the variance between developers. This handbook removes the variance.

Variance is the real risk.

The bad scenario isn't AI producing bad code. The bad scenario is five developers using AI in five different ways, producing five different qualities of code — and all of it shipping. One developer writes tests; another doesn't. One runs /code-review before every PR; another never has. The team's worst output becomes the team's average. Variance is the failure mode, not capability. The handbook eliminates it.

Skills are the operating system. The handbook is the user manual.

Our 101 skills define how a single piece of work should be done — how to plan a feature, run a code review, write a migration, ship a release. The handbook defines when each skill applies, who runs which one at which seniority, and what the team agrees not to do. Without the handbook, skills get used differently per developer, and we're back to variance. With it, the skills become the team's shared muscle memory.

This is shareable with clients.

Most engineering playbooks are internal — locked behind SSO, written for insiders, full of inside jokes and tribal knowledge. Ours is public by design. Clients should see exactly how we operate before they sign. Process transparency is itself a credibility signal. If we do something internally that we wouldn't tell a client about, it isn't worth doing.

Cornerstone

Four statements the handbook is built on.

Every page that follows is downstream of these. If a policy ever conflicts with one of them, the policy loses.

Statement 01

"AI is not a productivity multiplier unless you remove the variance between developers."

Statement 02

"A Software Engineer I with skills produces SE III-grade output. A Software Engineer IV without skills produces SE II-grade output. Tooling has flattened the curve — that's our advantage."

Statement 03

"We measure AI adoption by output consistency, not by speed."

Statement 04

"Our team produces the same quality regardless of who runs the keyboard. That's the standard. Everything else is process to get there."

We don't use AI for novel architecture decisions — hallucination risk exceeds productivity gain. We don't accept SE I pull requests without /code-review output attached; they're rejected at first review. The statements above are not aspirations. They are descriptions of the current floor.

Reader's Guide

Where to start, based on who you are.

The handbook is read top-to-bottom, but most readers don't need the whole thing on day one. Pick your starting point below.

Software Engineer I New to Heaptrace · First 90 days
Read Principles → How We Work § SE I → Anti-Patterns. In that order. The Anti-Patterns doc is the one your first code review will reference.
Software Engineer II Productive contributor · Building features end-to-end
Read Principles → How We Work § SE II → Skills Playbook. The Playbook is where the daily wins are.
Software Engineer III / IV Senior / Staff · Reviews, leads design, owns systems
Read Principles → How We Work § SE III/IV → Quality Gates. Quality Gates is the doc you enforce on everyone else.
Tech Lead / Engineering Manager Steward of the standard
All seven docs. You don't just follow the handbook — you're the reason it stays current. Quarterly review is on you.
Client / Prospect Evaluating how Heaptrace operates
Start with the Client Edition (Doc 07). It's the same content as the internal handbook, written for someone outside the team. No password required.
