## Phase 1: Setup вҖ” Prepare Your Workspace

Before writing any feature code, set up the environment.

### 1.1 вҖ” Create a Branch

```bash
# Create a feature branch from latest main
git checkout main
git pull origin main
git checkout -b feature/your-feature-name
```

**Branch naming:** `feature/<short-kebab-case-name>` (2-4 words)

```
feature/user-invitations     вң…
feature/course-certificates  вң…
feature/fix-stuff            вқҢ (too vague)
feature/new                  вқҢ (meaningless)
```

### 1.2 вҖ” Check Dependencies

## Phase 2: Backend вҖ” Database, API, Logic

Build from the bottom up: data layer first, then API, then business logic.

### 2.1 вҖ” Database Schema Changes

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ                 DATABASE CHANGE FLOW                    в”Ӯ
в”Ӯ                                                         в”Ӯ
в”Ӯ  Update Schema  в”Җв”Җв–¶  Create Migration  в”Җв”Җв–¶  Generate    в”Ӯ
в”Ӯ  (model file)        (migration tool)       Client      в”Ӯ
в”Ӯ                                                         в”Ӯ
в”Ӯ  вҖў Add new models    вҖў Run migrate dev     вҖў Regenerate в”Ӯ
в”Ӯ  вҖў Add fields        вҖў Name descriptively  вҖў Type-safe  в”Ӯ
в”Ӯ  вҖў Add relations     вҖў Review SQL output   вҖў Ready      в”Ӯ
в”Ӯ  вҖў Add indexes                                          в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

**Rules for schema changes:**
- Always add **reverse relations** on related models
- Use **descriptive migration names** (e.g., `add-user-invitations-table`)
- Add **indexes** for fields you'll query frequently
- Set **sensible defaults** for new columns

**Checklist:**
```
в–Ў Model added/updated in schema file
в–Ў Reverse relations added to all related models
в–Ў Indexes added for frequently queried fields
в–Ў Migration created and applied
в–Ў Client/types regenerated
в–Ў Verify: no errors when querying new model
```

### 2.2 вҖ” API Endpoints

Build endpoints in this order:

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ               API IMPLEMENTATION ORDER                   в”Ӯ
в”Ӯ                                                          в”Ӯ
в”Ӯ  1. Route definition     вҶ’ Define URL + HTTP method      в”Ӯ
в”Ӯ  2. Input validation     вҶ’ Validate request body/params  в”Ӯ
в”Ӯ  3. Auth & permissions   вҶ’ Check JWT + role/permission   в”Ӯ
в”Ӯ  4. Service/logic layer  вҶ’ Business logic in a service   в”Ӯ
в”Ӯ  5. Database operations  вҶ’ CRUD via ORM/query builder    в”Ӯ
в”Ӯ  6. Response formatting  вҶ’ Consistent response shape     в”Ӯ
в”Ӯ  7. Error handling       вҶ’ Proper status codes + messagesв”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

**For each endpoint, implement:**

| Layer | What to Build | Example |
|-------|--------------|---------|
| **Route** | URL path + method | `POST /api/v1/invitations` |
| **Validation** | Input schema | `{ email: string, role: enum }` |
| **Auth** | Permission check | `requireRole('admin', 'manager')` |
| **Service** | Business logic | `invitationService.create()` |
| **DB Query** | Data operation | `db.invitation.create({ data })` |
| **Response** | Output format | `{ id, email, status, createdAt }` |
| **Errors** | Error cases | 400 (invalid), 403 (forbidden), 409 (duplicate) |

### 2.3 вҖ” Business Logic

Keep logic in **service files**, not in route handlers.

```
Route Handler (thin)          Service (all logic)

в”Ӯ Parse request    в”Ӯ          в”Ӯ Validate business rules  в”Ӯ
в”Ӯ Call service     в”Ӯв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶в”Ӯ Check permissions        в”Ӯ
в”Ӯ Send response    в”Ӯ          в”Ӯ Execute DB operations    в”Ӯ
в”Ӯ                  в”Ӯв—Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ӯ Handle side effects      в”Ӯ
в”Ӯ                  в”Ӯ          в”Ӯ (email, cache, events)   в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ          в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

**Why:** Services are reusable, testable, and keep routes clean.

### 2.4 вҖ” Backend Verification

Before moving to frontend, verify the backend works:

```
в–Ў API endpoint responds correctly (use curl, Postman, or test)
в–Ў Validation rejects bad input with clear errors
в–Ў Auth blocks unauthorized access
в–Ў Database records are created/updated correctly
в–Ў Error cases return proper status codes
в–Ў No console errors or warnings
```

**Output:** Working API endpoints with validation, auth, and error handling.

```
в”Ӯ  2. Build new components       вҶ’ Small, focused, reusableв”Ӯ
в”Ӯ  3. Compose into page          вҶ’ Layout + components     в”Ӯ
в”Ӯ  4. Add state management       вҶ’ Local or global state   в”Ӯ
в”Ӯ  5. Wire to API                вҶ’ Fetch, mutate, cache    в”Ӯ

в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

### 3.2 вҖ” Before Building UI

**Always check existing components first.** Search the codebase:

```
Do I need...
  A modal/dialog?      вҶ’ Check for existing Dialog component
  A data table?        вҶ’ Check for existing Table component
  A form input?        вҶ’ Check for existing Input/Select components
  A multi-select?      вҶ’ Check for existing ComboBox/MultiSelect
  A confirmation?      вҶ’ Check for existing ConfirmDialog
  A status badge?      вҶ’ Check for existing Badge component
```

**Rule:** Wrap existing components, don't rebuild them.

### 3.3 вҖ” Build Components

For new components:

| Principle | Do | Don't |
|-----------|-----|-------|
| **Single responsibility** | One component = one job | A component that does 5 things |
| **Props over hardcoding** | `<Card title={title}>` | `<Card>` with hardcoded title |
| **Composition** | `<Dialog><Form>...</Form></Dialog>` | One mega component |
| **States** | Handle loading, error, empty, success | Only happy path |

### 3.4 вҖ” Handle All UI States

```
в”Ӯ                 UI STATE MATRIX                    в”Ӯ
в”Ӯ                                                    в”Ӯ
```

Before committing, review your own code:

```

```


## Tips for Best Results

1. **Follow the plan** вҖ” If you used `feature-plan` first, follow it. Don't improvise mid-build.
2. **Build backend first** вҖ” Always have a working API before building UI. Mock data slows you down.
3. **Check existing code** вҖ” Before building anything, search the codebase. Someone may have built something similar.
4. **Test as you go** вҖ” Don't save all testing for the end. Verify each phase before moving to the next.
5. **Small commits** вҖ” Commit after each logical chunk, not one giant commit at the end.
6. **Ask when stuck** вҖ” If something isn't clear, ask before building the wrong thing. 10 minutes of clarification saves hours of rework.
